mistune directive 默认使用反引号和波浪线做分隔符，如果想要自定义分隔符，可以这样：

```python
import mistune
from mistune.directives import FencedDirective, Figure

markdown = mistune.create_markdown(
    renderer=mistune.HTMLRenderer(),
    plugins=[
        FencedDirective([
            Figure(),
        ], marker=":")
    ]
)
```

如果要看一下产生的AST可以这样转为JSON：

```python
_, state = markdown.parse(figure_text)
tokens = json.dumps(state.tokens, indent=2, ensure_ascii=False)
print(tokens)
```

这里 `FencedDirective` 负责协调解析 directives，它有一个 `DirectiveParser` 用于解析包裹在最外面的分隔符：

````markdown
<figure class="panel-directive">
</figure>
````

使用如下正则：

```python
r"^(?P<fenced_directive_mark>(?:`|~){3,})"
r"\{[a-zA-Z0-9_-]+\}"
```

这条正则匹配分隔符 `:::`，然后我们从中提取出`{directive}`之后到结尾的 `:::` 之前的内容，也就是这一部分：

```markdown
{directive}
your content
...
```

这一部分又通过另一个正则匹配：

```python
re.compile(
    r"\{(?P<type>[a-zA-Z0-9_-]+)\} *(?P<title>[^\n]*)(?:\n|$)"
    r"(?P<options>(?:\:[a-zA-Z0-9_-]+\: *[^\n]*\n+)*)"
    r"\n*(?P<text>(?:[^\n]*\n+)*)"
)
```

这名命名组 `type` 对应你在大括号中写的名字，例如 `figure`，用这个名字去找对应的解析函数，也就是你注册的解析函数。

当你需要自定义 directive 插件时，需要继承 `DirectivePlugin`，它是什么？它实际上应该属于 `BaseDirective` 这个类的一部分，但是分拆出去以方便用户自定义。至于 `BaseDirective` 则扮演者在你的 directive 插件和 mistune 核心功能之间协调者的橘色。它把自己作为插件注册到 mistune，同时又把你自定义的解析逻辑整合到自己内部。

以 `figure` 插件为例，对于如下markdownn文本：

````markdown
<figure key="value" class="panel-figure">

<p class="panel-title">

src.png

</p>


This is caption

</figure>
````

首先 `parse` 方法接收到从`{figure}` 开始直到 `...caption` 的匹配。把上方的键值对解析成字典，图片解析成 `block_image`，底部的文本则调用 mistune 的块级解析器正常解析，检查第一个 token 是否是 `paragraph` 类型，如果是，则更该类型为 `figcaption`，剩余的token 则归为 `legend` 类型。

## 为什么要费劲自定义这种 directive？

markdown 支持直接写 html，所以，直接写

```html
<figure>
    <img src="src.png">
    <figcation></figcation>
</figure>
```

不是更好吗？

如果仅仅是为了“写文章”，写不出来的东西直接上 HTML，或者用 Hugo 的 Shortcode，简单粗暴又有效。
从纯粹的内容创作者角度来看，费劲去折腾解析器确实有点“没事找事”。

从写作/输出的角度确实没什么用，因为其实只要写的流畅，用word、pages、记事本或者手机的 Notes，都一样。但是如果我们把markdown当作数据容器，那就不一样了。传统的 SQL 作为容器时，可能更适合用于高度结构化的数据，比如交易记录，并且 SQL 不适合人机协作。

Markdown 是“半结构化”数据的完美载体。
现实世界中，80% 的数据是“半结构化”的——比如一篇技术文章，它有标题、段落（这是结构），但段落里夹杂着代码、图片、表格，甚至还有你自定义的 `:::{figure}` 属性。用 SQL 存这些，得拆成七八张表，极其痛苦；用纯文本存，又丢了结构。Markdown 正好卡在中间：既有文本的灵活，又有标记的结构。

同时，markdownn 是“人机协作”的最佳界面
对人来说，Markdown 是直观的，哪怕没有任何工具，用记事本打开也能读、能改。
对机器来说，Markdown 是有规律的，通过正则、AST、状态机，程序可以轻易地提取出里面的图片路径、元数据（id、class）、甚至是业务逻辑（比如某些特定的指令块）。
这种“人写起来不累，机器读起来不晕”的特性，正是现在很多现代技术栈（比如 Next.js 的 Contentlayer、Vue 的 VitePress，甚至很多 AI 训练数据的清洗管线）拥抱 Markdown（或 MDX）的原因。

此外，如果你的输出目标不是html，而是 LaTeX、Typst，在markdown中写html就不合适了。

这个问题背后还藏着一个计算机科学里非常经典的哲学博弈：“图灵完备的混乱” VS “受限语法的秩序”。
