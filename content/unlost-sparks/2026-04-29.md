## 架构

markdown-it-py 核心是传统解析器的lexer 和 parser 方法，只是它和传统的解析器在实现上有明显的不同。

传统的 Lexer -> Parser 是一个很明确的单向流动，所有的解析规则都在 Parser 类下， markdown-it-py 中 `MarkdownIt` 相当于 Parser，但是把解析规则分离出去了，而为了维持 Parser 拥有的唯一状态，引入了 `State` 把“解析规则（逻辑）”和“解析状态（数据）”彻底解耦了。

我们可以把 `MarkdownIt`、`Ruler` 和 `State` 的关系，用一种更形象的“剧组”比喻来理解：

1. MarkdownIt：它是整个解析器的总入口和总控中心。它不负责具体的搬砖干活，而是负责初始化各种解析器和渲染器、制定规则（启用或禁用哪些插件），并最终调用 `parse` 或 `render` 方法。
2. Ruler（规则管理器）：它手里拿着各种规则（Rule），并提前编译好了执行顺序和缓存（`__compile__`）。它只负责提供“该干什么活”的指令清单。
3. State：在传统的单体 Parser 中，解析到第几行、当前的 Token 列表是什么、原始文本是什么，这些变量通常都作为 Parser 的实例属性存在。但在 `markdown-it-py` 中，这些“正在发生的数据状态”被全部抽离出来，放进了 `State` 类里。

### 为什么要把状态抽离成 `State` 类？

这种设计带来了极大的灵活性和安全性，主要体现在以下两点：

**完美的递归与嵌套解析**：

Markdown 经常需要嵌套解析，
如果状态直接存在 Parser 实例上，一旦开始解析内部嵌套内容，外层的行号、指针（`pos`）就会被覆盖，导致解析错乱。
有了 `State`，每次解析（无论是块级 Block、行内 Inline 还是核心 Core）都会实例化一个全新的、独立的 `State` 对象。外层的 Parser 拿着外层的 Outer State），内层的内层的 Inner State，两者互不干扰，解析完再合并结果。

**规则函数的纯净与可复用**：

所有的解析规则（比如 `rules_block/heading.py`）都只是普通的函数。它们不依赖任何全局变量或类实例属性，只需要接收一个 `state` 参数，就能从中拿到文本（`state.src`）、当前解析位置（`state.pos` 或 `state.line`）和待填充的 Token 列表（`state.tokens`）。这让规则的编写、测试和插件化变得极其容易。

**总结一下 `markdown-it-py` 的流水线：**

当 `MarkdownIt.render()` 被调用时，它会先创建一个 `Core` 阶段的 `State`。这个 `State` 会像接力棒一样，依次经过：

1. Core 链：做全局预处理（如标准化换行符）。
2. Block 链：逐行扫描，把文本切分成段落、标题等块级 Token（此时 `State` 记录着行号 `line`）。
3. Inline 链：遍历块级 Token，为每个包含文本的块创建新的 `Inline State`，把文本解析成行内 Token（此时 `State` 记录着字符指针 `pos`）。

最后，这个装满了 Token 的 `State` 会被交给渲染器（Renderer），转换成最终的 HTML。

`MarkdownIt` 是总控，`Ruler` 是规则库，而 `State` 则是那个在规则之间流动、承载着所有解析进度和数据的“唯一状态容器”。这种设计模式在高性能解析器中非常经典（类似于编译原理中的 Context 或 Environment）。

## 缓存与函数的多对多映射

markdown-it-py 中的规则 rule 统一由一个叫 `Ruler` 的类管理：

```python
class Ruler(Generic[RuleFuncTv]):
    def __init__(self) -> None:
        # List of added rules.
        self.__rules__: list[Rule[RuleFuncTv]] = []
        # Cached rule chains.
        # First level - chain name, '' for default.
        # Second level - diginal anchor for fast filtering by charcodes.
        self.__cache__: dict[str, list[RuleFuncTv]] | None = None

    def __find__(self, name: str) -> int:
        """Find rule index by name"""
        for i, rule in enumerate(self.__rules__):
            if rule.name == name:
                return i
        return -1

    def __compile__(self) -> None:
        """Build rules lookup cache"""
        chains = {""}
        # collect unique names
        for rule in self.__rules__:
            if not rule.enabled:
                continue
            for name in rule.alt:
                chains.add(name)
        self.__cache__ = {}
        for chain in chains:
            self.__cache__[chain] = []
            for rule in self.__rules__:
                if not rule.enabled:
                    continue
                if chain and (chain not in rule.alt):
                    continue
                self.__cache__[chain].append(rule.fn)
```

这里的 `__compile__` 是在建立缓存，采用的是一种类似反响索引的方法，或者更简单滴说，就是数学课上讲解函数时的多对多映射。

我们用简化的代码做演示，假设 `Rule` 定义如下：

```python
class Rule:
    alt: List[str]

rule1 = Rule(
    alt: ['name1', 'name2']
)
rule2 = Rule(
    alt: ['name2', 'name3']
)
```

当你手中有了 `name2` 这个键的时候，怎么快速找到它所对应的所有规则呢？我们可以建立一个 `nameX` 到 `Rule` 的映射。也就是 `self.__cache__ = {}`，这与定义域和值域之间的多对多映射时相似的。

注意最初设置的这个键 `chains = {""}`，它实际上会包含所有的规则，`for rule in self.__rules__` 这个循环中有两次中断，第二次中断时的检测：`if chain and (chain not in rule.alt)` 如果这个键不是空并且不存在于 `rule.alt` 中，那么这个键就跳过，显然空键在这里判断永远是 `False`，所以每一条规则都会放入空键中，从而成为默认规则。

这里缓存的意义是什么？
如果不做这个 `__compile__`，每次 markdown-it 解析文档时，遇到一个需要处理的语法，都得把几百个规则从头遍历一遍，挨个检查 `if rule.enabled` 和 `if name in rule.alt`，效率会非常低。
通过 `__compile__` 提前建立好这个“反向索引/映射缓存”，解析器在运行时只需要直接通过链名（比如 `cache["emphasis"]`）瞬间拿到所有需要执行的函数列表，直接按顺序调用即可。这是一种典型的“空间换时间”的性能优化策略。
